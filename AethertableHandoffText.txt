DEVELOPER HANDOFF · v1
Aethertable
A tabletop-RPG session app — the complete UI, none of the wiring.

This document exists so that a developer who has never seen this app before can pick it up and know exactly what every screen and button is supposed to do, which pieces already work for real, which pieces are hollow mockups, precisely how each hollow piece was faked, and what to build to make it real. Read the first three sections before touching anything.

0 · How to read this guide

Every feature below is tagged with one of three labels. Wherever a thing is faked, the entry names the real behaviour first, then explains exactly how it was faked, then tells you what to build.

REAL
Actually implemented. It runs real logic (math, state changes) and does not need to be rebuilt — though most "real" pieces still act on placeholder data.
PARTIAL
The interaction works, but only against fake data or in a single hard-coded case. Real logic exists but is not connected to anything persistent or general.
FAKED
Visual only. Clicking it does nothing meaningful — usually it just shows a small notification ("toast") in the bottom-right corner so the screen feels alive during a demo. No data changes, nothing is saved.
THE ONE THING TO UNDERSTAND

There is no backend, no database, no accounts, and no networking anywhere in this app. Every character, monster, item, scene, and campaign you see is hard-coded fake data living in memory. When you reload the page, everything resets to its starting values (with one exception — see "Image drop-zones"). Making Aethertable real is, at its core, the job of replacing that in-memory fake data with a real server + database, and replacing the "toast" notifications with real actions.

1 · What Aethertable is (for someone who's never seen it)

Aethertable is a shared digital tabletop for playing Dungeons & Dragons–style role-playing games. In these games one person, the Game Master (GM), runs the world and story; the other people are Players, each controlling one hero character. Aethertable's guiding principle is controlled simplicity: the GM gets powerful world-building and control tools, while each player is shown only the few things that matter to them at that exact moment, so they are never buried in menus.

The app has two kinds of "scene," and the whole interface changes depending on which is active:

Playing Field — a relaxed scene for conversation, exploration, and shopping (a tavern, a market). It's just a location image with character markers ("tokens") on top. The GM moves everything; players mostly talk and make choices.
Battle Field — a tactical combat scene on a gridded map. Here players directly control their own token on their turn: move, attack, cast spells. The app also supports walls, lighting, and interactive objects like levers and traps.

A Campaign is the container for everything: all the scenes, plus the reusable library of characters, monsters, items, shops, images, music, and notes. The GM prepares scenes in Build Mode, then runs the live game in Session Mode.

You navigate the running mockup with the left sidebar and the two toggles in the top bar: Build / Run Session (GM only) and Game Master / Player (which swaps the entire interface between the two roles so you can preview both from one screen).

2 · How it's built & where the fake data lives

The entire app is a single file, Aethertable.dc.html. It is one component with two parts: a template (all the visible markup) and a logic class (all the data and behaviour). You do not need to keep this single-file structure — a real build would split it into proper components — but everything you need to understand is in that one file today.

All the fake data is defined in two obvious places inside the logic class, and this is where almost all of your "make it real" work connects:

Static content arrays (class fields near the top): campaigns, party, PCS, scenes, bestiary, npcs, itemsCatalog, tracks, mediaImgs, notesData, templatesData, players, interactiveObjects, sceneStates, promptsList, reinforcements, encounterEnemies, history, enemyAC. These are your "database tables," hard-coded. Replace every one of these with data fetched from your server.
Live game state (the state object): the current role, which screen is open, token positions, hit points, whose turn it is, gold, shop settings, and which panels are open. In a real product most of this becomes shared, server-synced state so every player sees the same board in real time.
THE BIGGEST MISSING PIECE: MULTIPLAYER

Right now the "GM / Player" toggle just re-skins one screen on one computer. In the real product the GM and each player are on separate devices, and the board, initiative, hit points, and reveals must stay in sync live across all of them. That real-time sync layer (websockets or similar) does not exist yet and is the single largest thing to build. Every "REAL" interaction below currently only updates the local screen; each one needs to broadcast its change to everyone else.

3 · What is genuinely working (NOT faked)

These pieces run real logic. They still operate on the placeholder data described above, and none of them sync across devices yet, but the behaviour is built and correct — you should not have to rewrite the logic, only connect it to real data and the network.

Dice rolling

I did not fake the dice. The "d20" button in the top bar, and every "HIT / DMG / ROLL / Cast" button on character sheets and monster stat blocks, generate real random numbers with real modifiers and report the result. Attack buttons roll a real d20, add the attack bonus, and announce the total (flagging natural 20s and 1s); damage buttons roll the correct dice (e.g. 2d6+3) and sum them. This is real and reusable — it just needs to be shown to the whole table instead of only the roller.

Combat attack resolution (player Battle Field)

When a player picks an attack and then a target during combat, the app really rolls to hit, compares the total against that enemy's Armor Class, and — on a hit — rolls damage and subtracts it from the enemy's hit points (the token can drop to "DOWN"). This is real logic. It reads the enemy's AC from the hard-coded enemyAC table, which you'd replace with real monster data.

Token dragging on the map

Every token on any map can be click-dragged to a new position, and its stored coordinates update for real. This is real. What's missing: it isn't broadcast to other players, there's no grid-snapping enforcement, and it doesn't check walls or movement distance.

Placing & erasing map geometry (Battle Field builder)

In Build Mode, choosing the Wall, Half-wall, Light, Object, or Token tool and clicking the map really adds that element at the clicked spot; the Erase tool really removes the nearest element. This is real. It's basic (walls are fixed-length; there's no drag-to-draw or endpoint editing yet), and the geometry isn't yet fed into any real line-of-sight calculation.

Hit-point changes

Every "+/– HP", "Damage/Heal", healing potion, and Second Wind button really changes the stored hit points, and every health bar (on tokens, initiative, sheets, the party list) reflects it live and re-colours green→amber→red. Real. Not yet synced across devices.

Turn order (initiative)

In a Battle Field session, "Next turn / Prev turn" really advance through the initiative list, highlight the active combatant, and increment the round counter when the order wraps around. "Add / Summon" really inserts a new combatant into the order. Real. The list itself is fake starter data.

Buying from a shop & spending gold

In the player storefront, "Buy" really checks whether the party has enough gold, and — when the shop is in "Automatic" mode — deducts the price, adds the item to the character's inventory, and blocks the purchase if gold is short. It also respects the GM's chosen transaction mode: in "Approval" mode it says a request was sent; in "Manual" mode it tells the player to ask the GM. This purchase logic is real. What it lacks is a real transaction/approval round-trip to the GM.

Every toggle, tab, slider, and navigation control

All screen navigation, the role/mode switches, the character-sheet context tabs, the shop price slider and rules switches, the grid-size slider, the time-of-day / weather buttons (which really re-tint the map), the "Show grid / Fog of war / Dynamic lighting" switches (which really show/hide those layers), music play/stop, and hiding/showing shop items — these all really change what's displayed. What they change is only the local view of fake data.

Image drop-zones

Every "drop an image here" placeholder (campaign covers, scene thumbnails, portraits, map backgrounds, the media library) is a real drag-and-drop target: drop an image file and it sticks, even across reloads. This is the only thing that persists between reloads. It's a mockup convenience, not a real asset pipeline — a production build needs proper uploads, storage, and a media CDN.

4 · Everything that is faked, area by area

A note that applies everywhere below: many buttons trigger a small notification in the bottom-right corner (a "toast," e.g. "Scene saved" or "Reinforcements triggered"). These toasts are the fake. They exist so the app feels responsive in a demo, but behind each one nothing is created, saved, sent, or changed. Wherever an entry says "shows a toast," read that as "does nothing real yet."

A · Accounts, sign-in & the whole shell

Sign-in / accounts FAKED (absent)
A real product needs users to log in and have identities so the app knows who is the GM and who each player is. There is no sign-in at all — the app opens straight into a session as a pre-set "GM" user (the "GM" avatar top-right is decorative). To build: add authentication and a user/account model; the top-right avatar and the "Viewing as" sidebar card should reflect the logged-in user.

Game Master ↔ Player toggle PARTIAL
This is meant to represent two different people on two different devices. It really works as a preview: flipping it instantly rebuilds the interface with the correct role's tools and sidebar. But it's a local disguise, not two real sessions. To build: role comes from the logged-in user; the GM and players connect from separate devices to a shared live session.

Build / Run Session toggle REAL
Really switches the GM between the scene-editing interface and the live-play interface. Real behaviour; nothing to fix beyond wiring it to a real, running session.

Player join code + "COPY" + "4 players connected" FAKED
In the real product this is the code a player enters to join the GM's live game, and the count shows who's actually connected. The code (EMBER-7429) is a hard-coded string, "COPY" shows a toast (it doesn't touch the clipboard), and "4 players connected" is static text. To build: generate a real per-session code, wire "COPY" to the clipboard, and drive the count from real connections.

Settings (gear, bottom of sidebar) PARTIAL
Opens a real Settings screen (see section N). Navigation is real; the controls on it are mostly faked.

B · Campaigns hub (the opening screen)

What it is: the list of game-worlds you can enter. Each card is one campaign.

The campaign cards FAKED
Four campaigns are shown; they are fixed entries in the campaigns array (name, system, player/scene/session counts, "last played" text, and the green "LIVE" badge are all invented). Clicking any card opens the same one demo campaign's Overview — it does not actually load different campaigns. To build: fetch the signed-in user's real campaigns; each card opens that specific campaign.

"＋ New campaign" FAKED
Should create a brand-new empty campaign. Shows a toast only. To build: create a real campaign record and open it.

"Join with code" + the code field + "Join" FAKED
Should let a player type a GM's code and join that live game. The panel really opens and you can type, but "Join" only shows a toast; nothing is validated or joined. To build: look the code up on the server and connect the player to that session.

C · Campaign Overview (the dashboard)

What it is: the campaign's home page — a snapshot of the saved world and recent play.

"Saved world state" tiles & the status list FAKED
This is meant to be the real, automatically-saved state of the game: where the party is, how many enemies are alive, which doors are open, gold on hand, and story flags (shard recovered, boss captured, etc.). Every number and line here is hard-coded (stateStats and worldFlags) and "AUTO-SAVED · 2 MIN AGO" is decorative. To build: derive all of this from the actual live game state and a real save system.

"The party" cards PARTIAL
The four heroes are fake (the party / PCS data), but their health bars are driven by the same real HP values used everywhere, and clicking a card really opens that character's sheet. To build: real characters from the database.

"Resume session" / "Session history → ALL" PARTIAL
These really navigate (to the live session, and to the History screen). The history entries themselves are fake (history array).

D · Scenes library

What it is: all the GM's prepared locations and encounters, filterable by folder (Locations, Chapters, Encounters, Completed, Unused ideas).

The scene cards & folder filters PARTIAL
The nine scenes are fake (scenes array). The folder-filter chips really filter the list (real), and each card's Playing/Battle badge is real to its data. To build: real scenes from the database.

"Edit" and "▶ Run" on each card REAL
Really open that scene in the Build editor, or start it as a live session, and correctly load the Playing- vs Battle-Field interface for its type. Real navigation.

"⧉ Duplicate", "✕ Delete", "＋ Playing Field", "＋ Battle Field" FAKED
Should copy, remove, or create scenes. Duplicate and Delete only show a toast (the card is not actually copied or removed). The two "＋" buttons do open a blank editor of the right type, but nothing is saved when you leave. To build: real create/duplicate/delete against the database.

E · Build Mode — Playing Field editor

What it is: where the GM sets up a roleplay location — the background image, character tokens, environment, and the interactive bits players can be offered.

The map, tools, and token dragging REAL
Dragging tokens and placing tokens/highlights with the tools works for real (see section 3). The background is a real image drop-zone.

Time of day & Weather buttons REAL (as a visual tint)
Really re-tint the map (dusk = warm, night = dark blue, rain/fog add haze). This is a genuine visual effect, not a toast. It's a simple colour overlay, not a real lighting/weather simulation.

Selected-token panel: Expression / Pose / Show portrait / Highlight FAKED · Remove REAL
These are meant to change a character's on-screen face/pose, pop up a big dialogue portrait, or spotlight them for players. Because the tokens are simple coloured circles with no real artwork, expression/pose/portrait/highlight only show a toast. "Remove" genuinely deletes the token from the map. To build: give tokens real multi-image art sets so expression/pose can swap the shown image, and build a real dialogue-portrait overlay players see.

Ambient sound player PARTIAL
The play/stop button really toggles a "playing" state and updates the label, but no audio actually plays — there are no real sound files. To build: attach real audio files and play them for the whole table.

Visual states & transitions FAKED
Meant to let the GM flip the whole scene between prepared looks ("Peaceful evening" → "Bar brawl"). The three states are fake (sceneStates) and clicking one only shows a toast. To build: let a GM save multiple arrangements of a scene and switch between them.

Interactive prompts (list + "SEND" + "＋ ADD") FAKED
A prompt is a little choice the GM pushes to players ("Inspect the statue"). The prompts are fake (promptsList); "SEND" and "ADD" show toasts and never reach a player. To build: real prompt authoring, and delivery to players' screens with their responses coming back.

"Manage storefront" PARTIAL
Really opens the shop-manager panel (see section L).

F · Build Mode — Battle Field editor

Grid controls, vision/lighting switches, wall/light tools REAL (visually)
"Show grid" and the cell-size slider really draw/resize the grid overlay on the map; the Wall/Half-wall/Light/Object tools really place geometry and Erase really removes it (section 3). "Snap to grid," "Dynamic lighting," "Line of sight," and "Fog of war" switches really flip on/off and Fog really shows/hides a darkened overlay. The important caveat: these are visual layers only. There is no real line-of-sight or lighting engine — walls don't actually block any player's vision, and fog is a fixed decorative gradient, not a per-player computed reveal. To build: a real vision system that computes what each character can see from their token, using the walls, and reveals/hides the map per player.

Interactive objects list ("Iron Gate," "Lever A," etc.) FAKED
These represent map objects with real gameplay effects (a lever opens a gate, a barrel explodes). The list is fake (interactiveObjects); each object's "state" and description are text. "EDIT LOGIC" opens the trigger editor (section G); "＋ ADD" just selects the Object placing tool. To build: attach real behaviour to placed objects.

Encounter & reinforcements list FAKED
The planned enemies and their "HIDDEN" flags, plus the reinforcement triggers ("After round 2"), are fake data (encounterEnemies, reinforcements) shown for reference. To build: real enemy placement and rules that actually spawn reinforcements when their condition is met.

"Save" & "▶ Run scene" PARTIAL
"Run scene" really launches the live session. "Save" only shows a toast — edits are not persisted. To build: persist the scene.

G · The Trigger Editor

What it is: a visual, no-code way for a GM to chain events — e.g. "When Lever A is pulled → Iron Gate opens → Alarm activates → after 2 rounds, Guards enter."

The whole editor FAKED
It opens for real and looks complete: a step-by-step chain plus palettes of triggers ("Lever pulled," "HP threshold") and effects ("Open door," "Spawn enemies"). But the chain shown is a fixed example (triggerFlow), the palette chips only show a toast when clicked, "＋" and "Save rule" do nothing real, and — most importantly — none of these rules actually run during a session. To build: a real rules engine that stores trigger→condition→delay→effect chains and executes them live when the trigger fires.

H · Session Mode — GM running a Playing Field

Selected-NPC controls, Time/Weather, Ambient, Present-a-prompt MIXED
Same pieces as the Playing-Field builder: Time/Weather really re-tint the map (real); ambient toggles state but plays no audio (partial); expression/portrait and "present a prompt" show toasts (faked — need real art, real prompt delivery). Selecting a token and dragging it is real.

Reveal…, Whisper…, Secret d20 (bottom-left of the map) MIXED
"Secret d20" really rolls a private die (real). "Reveal…" and "Whisper…" open real panels (section I) whose contents are faked.

I · Session Mode — GM running a Battle Field & "Play God"

Initiative rail + Next/Prev turn REAL
Turn advancement, round counting, current-combatant highlight, and the live health bars are all real (section 3). The combatant list is fake starter data.

Selected combatant: "– 5 HP / + 5 HP" REAL · "Add condition" FAKED
The HP buttons really change that token's health. "Add condition" (meant to apply a status like Poisoned or Prone) only shows a toast. To build: a real conditions system that tracks statuses and their effects.

Hazards & Objects "TRIGGER", Reinforcements "SUMMON" MIXED
"TRIGGER" is meant to fire an object's effect (spring the trap, open the gate); it only shows a toast (faked). "SUMMON" actually adds a combatant to the initiative order (real), but doesn't place a matching token on the map. To build: connect object triggers to real effects; make summon also drop the token on the board.

Lighting & vision switches PARTIAL
Same as the builder: they flip the visual layers but there is no real vision engine behind them.

The "⚡ Play God" panel MOSTLY FAKED
This is the GM's override toolkit — the promise that automation never limits the GM. Two of its ten actions are real: "Add initiative entry" really inserts a combatant, and "Reveal/Conceal the map" really toggles the fog layer. The other eight (Move any token, Set HP/condition, Force/cancel effect, Undo, Rewind to round start, Ignore a rule, Custom outcome, Control a PC) only show a toast. To build: real implementations — most notably a proper undo/rewind history of the game state, and a way for the GM to temporarily drive a player's character.

J · Session Mode — a Player on a Playing Field

What it is: the deliberately simple player view — mostly watching and choosing. The bottom bar holds their portrait, sheet shortcut, roleplay actions, a dice request, and a shop button.

Roleplay action buttons FAKED
("Talk to the innkeeper," etc.) Represent actions the player declares to the table. They only show a toast; nothing is sent to the GM. To build: deliver the declared action to the GM's screen.

The "GM ASKS … Roll" request PARTIAL
Represents the GM asking a specific player for a check. The prompt text is hard-coded (the GM never really sent it), but pressing "Roll" performs a real dice roll with a real success/fail result. To build: let the GM actually send a check request that appears here, and return the result to the GM.

"CHARACTER SHEET ▸" REAL
Really opens the player's sheet in a context-appropriate view.

"🛍 Shop" REAL (opens) · see section L for the storefront itself.

K · Session Mode — a Player on a Battle Field

What it is: the player's tactical turn interface. On their turn they get an action bar; otherwise they see who's acting.

"Move" (distance preview) PARTIAL
Toggles a real dashed circle around the player's token showing roughly how far they can move. It's a fixed-radius indicator, not a true speed-and-terrain-aware movement calculation, and moving is still done by dragging the token. To build: compute the real reachable area from speed, difficult terrain, and walls.

Attack / Spell / Item / Ability menus & targeting REAL (against fake data)
This is the app's showcase of reading the character sheet to offer only the relevant actions. It's genuinely built: "Attack" lists the character's real weapons, "Item" lists usable items (the healing potion really heals), "Ability" lists real class features (Second Wind really heals), and after choosing an attack you pick a target and the app really resolves the hit and damage (section 3). "Spell" correctly shows that this particular demo character (a Fighter) has no spells. All of it reads from the fake PCS sheet data. To build: feed it real character data; the action/targeting/resolution logic itself is done.

"End turn" REAL · "Ask GM" FAKED
"End turn" really advances initiative. "Ask GM" (for requesting an unusual action) only shows a toast. To build: real player→GM messaging.

"Whose turn" waiting state REAL
When it isn't the player's turn, it correctly names the current combatant and hides the action bar. Real, driven by the real initiative state.

L · Shops — the player storefront & the GM shop manager

Player storefront (buying) REAL (against fake data)
Lists the shop's visible items with prices adjusted by the GM's discount, shows the party purse, marks items you can't afford, and "Buy" runs real purchase logic that respects the GM's transaction mode (section 3). The item list is the fake shopStock. To build: real items, and a real approval round-trip for "Approval"/"Manual" modes.

GM shop manager panel MOSTLY REAL (locally)
Almost every control here really changes the shop's behaviour on this device: the transaction-mode selector (Automatic/Approval/Manual) really governs how "Buy" behaves; the price slider really marks all prices up/down and the storefront reflects it; the "Hide from players" toggle on each item really removes it from the player storefront; the rule switches and "Visible to" audience selector really toggle. What's faked: "＋ ADD ITEM" shows a toast, the audience selector doesn't actually build different lists per player, and none of these settings are saved to a server. To build: persistence, real per-player inventories, and real "bargaining" / "require a persuasion check" flows (currently just on/off switches with no effect on a purchase).

M · Character sheets

The sheets & the Full/Combat/Social/Shopping tabs REAL (against fake data)
All four heroes have complete sheets (abilities, skills, saves, AC, speed, attacks, spells & slots, features, conditions, inventory, currency, personality) in the PCS data. The context tabs really show/hide the sections relevant to each situation — a real demonstration of the "only show what matters now" idea. All the numbers (skill modifiers, saves) are really computed from the ability scores. To build: real, editable character data from the database.

Roll buttons (HIT / DMG / CAST), Damage/Heal REAL
Real dice and real HP changes (section 3). "CAST" currently just announces the spell (a real spell-effect system would go further).

"＋ add" condition, resource pips, spell-slot boxes FAKED (display)
Conditions, resource dots (Second Wind, etc.), and spell-slot counts are shown from fake data and don't decrement when used from the sheet; "＋ add" shows a toast. To build: make these track and spend real resources.

6 · Suggested order to make it real
Accounts & data: add sign-in and a database; replace the section-5 arrays with real records. This alone makes most screens show real content on top of the already-working controls.
Real-time session sync: put the live game state (board, HP, initiative, reveals) on a server and sync it across devices. This is what turns the "GM/Player toggle" into two genuine roles on two devices.
Persistence of edits & saves: make "Save," scene create/duplicate/delete, and the auto-saved world state real, plus session logging and restore.
Player→GM messaging: whispers, check requests, "Ask GM," roleplay actions, prompts, and shop approvals — the two-way traffic that's currently just toasts.
The vision/lighting engine: make walls actually block sight and fog reveal per-player. This is the hardest single system and can come last.
The rules/trigger engine & conditions: make the trigger editor's chains, interactive-object effects, and status conditions actually execute during play.
Real media & art: audio playback, token art sets (for expressions/poses), and dialogue portraits.

Good news for whoever picks this up: the hard, fiddly game logic — dice, attack resolution against AC, hit-point tracking and health bars, initiative and rounds, the sheet-driven action menus, gold-and-purchase math, and every navigation/toggle interaction — is already built and correct. The bulk of the remaining work is plumbing: a backend, real-time sync, and replacing "toast" stand-ins with real actions.
